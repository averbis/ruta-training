Introduction

About the Exercises

  • The exercises are grouped into unit with increasing degree of complexity. The reader is encouraged to practise in the order they are given in this document with one exception. The exercises 1-3 of Unit III about the UIMA Ruta Workbench should be preponed if the reader is not yet familiar with the usage of the tooling.

  • Some exercises of this document require a certain degree of familiarity and prior knowledge about UIMA, regular expressions, Eclipse and Maven. While several exercises link to important information about UIMA Ruta, missing knowledge about other technologies needs to be researched by the reader.

Sources of Information for UIMA Ruta (increasing degree of detail)

  • The UIMA Ruta homepage provides an overview of the system and links to additional sources of information.

  • The Section Core Concepts in the documentation contains a concise summary of the main ideas.

  • The Section Installation provides information how to install the UIMA Ruta Workbench. Screencasts are available for installing Eclipse and for installing the UIMA Ruta Workbench .

  • The tutorial slides of the GSCL 2013 provide an overview of the language and tooling (UIMA Ruta 2.2.1).

  • The main article about UIMA Ruta provides a 30-page summary.

  • Finally, the documentation contains the reference of the language, components and workbench.

Unit I: Language Elements

This unit provides some basic exercises for the UIMA Ruta rule language introducing its language elements. It is not intended to serve as a documentation or tutorial. The reader should be familiar with the following Sections from the documentation:

  • Core Concepts

  • Learning by Example

  • Syntax

  • Rule elements and their matching order

  • Basic annotations and tokens

  • Robust extraction using filtering

Other Sections of the documentation can be used as a reference if specific information about certain language elements is needed.

The workspace provides a project for each exercise of this unit. These project should be utilized to create the solution for the exercises since they contain additional material, e.g., the input documents.

Exercise 1: Basic Annotations

  • Declare a new annotation type "Animal".

  • Annotate an occurrence of an animal by checking the covered text of an annotation.

  • Annotate an occurrence of an animal by using a wordlist.

  • Annotate an occurrence of an animal with a simple regexp rule.

Exercise 2: Sequential Patterns

  • Declare a new annotation type "AnimalEnum".

  • Annotate enumerations of animals using a composed rule elements and the quantifier "+" or "*".

  • Declare a new annotation type "Sentence".

  • Create a "Sentence" annotation for each sentence in the input document. The last line may be ignored.

  • Create a "Sentence" annotation for each sentence in the input document with only a single rule using a wildcard. The last line may be ignored.

  • Declare four new annotation types: "Day", "Month", "Year" and "Date".

  • Create a single rule for detecting dates of the form "DD.MM.YYYY". This single rule must create four annotations. A "Date" annotation for the complete date mention. A "Day" annotation for the two digits of the day. A "Month" annotation for the two digits of the month. A "Year" annotation for the four digits of the year.

Exercise 3: Complex Annotations

  • Declare a new annotation type "EmplRelation" with the two features "employee" and "employer" of the type annotation.

  • Create an "EmplRelation" annotation and fill the features using the CREATE action.

  • Create an "EmplRelation" annotation and fill the features using the GATHER action.

  • Create an "EmplRelation" annotation and fill the features only using implicit actions.

  • Create an "EmplRelation" annotation and fill the features using annotation variables.

  • Create an "EmplRelation" annotation and fill the features using labels for the rule elements.

  • Declare a new annotation type "MoneyAmount" with an INT feature "amount" and an STRING feature "currency".

  • Create annotations for mentions of amounts of money and fill the features with correct values.

Exercise 4: Conditions and Actions

The project of this exercise already contains some rules for creating annotations, which can be utilized in the following.

  • Annotate all sentences that contain a number or a comma.

  • Annotate all sentences that contain at least two capitalized words.

  • Annotate all sentences that start with an animal.

  • Remove all sentences that contain an animal.

  • Remove all amounts of money whose amount is less than 50.

  • Remove all amounts of money whose currency is dollar.

  • Expand the "EmplRelation"" annotation to the complete document.

  • Trim the sentences by punctuation marks.

  • Split the sentences by the word "are".

Exercise 5: Advanced Structures

The project of this exercise already contains some rules for creating annotations, which can be utilized in the following.

  • Annotate all pairs of tokens within sentences (not across sentences) using a BLOCK construct.

  • Annotate all pairs of tokens within sentences (not across sentences) using an in lined rule.

  • Annotate all sentences that contain an EmplRelation with an employee named "Peter".

  • Annotate all sentences that are followed by a sentence that contains an MoneyAmount in Euro.

  • Create a new script, import and call the existing script.

  • Create a new script, import and call only the BLOCK construct of the existing script.

  • Create a new script, import the existing script and apply it only on capitalized words.

  • Annotate each word that is either followed or preceded by a small-written word using a FOREACH block.

Exercise 6: Visibility

The project of this exercise already contains some rule for creating annotations, which can be utilized in the following.

  • The PlainTextAnnotator created Line annotations. Try to create a sentence annotation for each Line annotation, then recap the visibility feature of UIMA Ruta.

  • Change the filtering setting so that Sentence annotations are created for all Lines.

  • Make capitalized words invisible and annotate all matchable sentences.

  • Create an annotation covering the text passage "Peter works for Frank. 10€" with the rule "(CW NUM SPECIAL){-> Type};"

Exercise 7: Advanced Concepts

  • Import TypeSystem1.xml and TypeSystem2.xml with aliasing.

  • Annotate the three tokens before a number with type "Type" from typesystem TypeSystem1.xml

  • Annotate the next period before a number with type "Type" from typesystem TypeSystem2.xml

  • Optimize the execution efficiency of the previous rules by using an anchor "@".

Unit II: Specific Tasks

The workspace provides a project for each exercise of this unit. These project should be utilized to create the solution for the exercises sine they contain additional material, e.g., the input documents.

Exercise 1: Phrase Chunking

The goal of this exercise is the create a minimal chunker, a component that annotates noun, verb, and prepositional phrases. Given a reduced set of Part-of-Speech annotations, annotations of the types ChunkNP, ChunkVP and ChunkPP should be created. For simplicity reasons, the rule script of the project creates some mockup Part-of-Speech tags.

Exercise 2: Enumerations

The goal of this exercise is to create a simple script for annotating enumerations. Declare a new annotation type "Enumeration" with one feature "elements" of the type "FSArray". Specify rules so that the annotations of the type "Enumeration" cover the complete span of the enumeration in the examples while their features contain the single elements of the enumerations. It is allowed to either integrate a Part-of-Speech tagger or simulate its integration like in the previous exercise..

Exercise 3: Sectioning

The goal of this exercise if to create a simple rule script for segmenting documents by their headlines. The resulting annotation named "Section"" should contain two features "headline" and "content". The "headline" feature should contain an annotation of the type "Headline", which specifies the headline of the section. The "content" feature should contains an annotation of the type "Content", which specifies the content of the section. The offsets of the "Section" annotation should cover the headline as well as the content. Declare the required types and specify rules that process all input documents of the project as correct as possible. Subsections, listings, tables, figure, header and footers do not need to be considered, but may serve as a voluntary extra task.

Exercise 4: Numerical Values

The goal of this exercise is to create a script for detecting and normalizing numerical values like "1/4" or "thirty one". Declare a new annotation type named "NumericValue" with a feature "value" of the type DOUBLE. The project contains an exemplary input document, which contains two numerical expressions in each line. The first one represents the text passage that should be annotated with "NumericalValue" and the second one represents the value of feature "value". Try to annotate as many as possible with the correct feature value. The sections about number expressions , PARSE condition and word tables may help in one or two examples. Another hint: the simple regexp rules can be utilized to create annotations within tokens.

Exercise 3: Undesirable Effect Relations in Tables

The goal of this exercise if to create a simple rule script for extracting undesirable effect information from tables. Declare a new annotation type named "UndesirableEffect" with three features of the type Annotation. The first feature named "class" represents the symptom class of the undesirable effect. The second feature named "effect" represents a actual effect of the relation. The third feature named "frequency" represents the frequency of the undesirable effect. The project contains some exemplary input documents. The first relation in example1.txt is: class: "Metabolism and nutrition disorders", effect: "diabetes mellitus", frequency: "very common". The script in the project already includes the HtmlAnnotator, which creates annotations for rows (TR) and cells (TD). Specify a set of rules as generic as possible that extract all undesirable effect relations.

Note

Information about the Ruta Projects can be found in the documentation

  1. Create a new Ruta Project named "MyFirstProject".

  2. Create a package in the script folder named "uima.ruta.training".

  3. Create a script file in the new package named "MyFirstScript".

  4. Declare some types and add some rules to the script and save the script file.

  5. Investigate the files "MyFirstScriptEngine.xml" and "MyFirstScriptTypeSystem.xml" in the descriptor folder "descriptor/uima/ruta/training/".

  2. Investigate the results in the views "True Positives", "False Positives", "False Negatives",

  3. Launch Main.ruta in debug mode (button with bug left to run button) and investigate files in output folder.

Exercise 4: Ruta in Maven Projects

Note

Information about the Ruta Maven Plugin can be found in the documentation . The project ruta-clinical-trials is a maven-built project and can be used as an example.

Create and configure a new maven project with following characteristics:

  1. The project contains two script files, whereas the first one imports and calls the second one.

  2. The second script file uses a compiled wordlist for creating annotations for a newly defined type.

  6. All required sources and resources are included in the resulting jar artifact.

Unit IV: Segmentation of References

The goal of this unit is to develop a set of rules for segmenting scientific references according to their bibtex fields . The rules have to create annotations of a reduced set of four different types: Author, Title, Date represent the main information for distinguishing publications. Additionally, Venue represents all remaining parts of the reference like Booktitle or Volume. The annotation schema is illustrated in the following using inlined xml elements for representing the corresponding annotations: 

<author>Katchalski-Katzir E, Shariv I, Eisenstein M, Friesem A, Aflalo C, 
Vakser I:</author> <title>Molecular surface recognition: determination 
of geometric fit between proteins and their ligands by correlation 
techniques.</title> <venue>Proc Natl Acad Sci USA</venue> <date>1992,
</date> <venue>89(6): 2195-2199.</venue>

A preconfigured ruta project named ruta-references is included in the workspace and may serve as a starting point. The project contains the type system, an empty script and the first two parts of an annotated data set. Both parts can be used to develop the rules. A hidden third part will be applied for evaluating the resulting rules. Furthermore, the comprehensibility, runtime performance and maintainability of the defined rules are of importance. Only UIMA Ruta functionality may be utilized for solving this task, e.g., rules and wordlists. The usage of external components is not allowed.

Unit V: Endpoints in Clinical Trials

The goal of this unit is to create an application for extracting endpoint information from pubmed abstracts about clinical trials. Further information about clinical endpoints in general can be found in the wikipedia article . The targeted endpoints are restricted three main types with different manifestations: overall response rate (ORR), overall survival (OS) and progression free survival (PFS). The latter two are further distinguished in the median value, the time interval and its rate. The following table gives an overview over the annotations types that should be identified:

Table 1. Information to extract in this unit.

median OSorg.apache.uima.trials.type.OSMean... median OS was [12 months].
OS time spanorg.apache.uima.trials.type.OSTimeThe [one-year] survival rate was 55.8%.
OS rateorg.apache.uima.trials.type.OSRateThe one-year survival rate was [55.8%].
median PFSorg.apache.uima.trials.type.PFSMeanThe median progression-free survival (PFS) was [10 months].
PFS time spanorg.apache.uima.trials.type.PFSTime[12-month] PFS was 14.8%.
PFS rateorg.apache.uima.trials.type.PFSRate12-month PFS was [14.8%].

In this unit, it is allowed to utilize external components if available in order to solve the task. Useful components could include amongst others: Tokenizer, Sentence Splitter, Stemmer, Lemmatizer, POS Tagger, Chunker, Parser, Enumeration Detection, Numerical Value Annotator, Physical Unit Annotator.

A preconfigured maven project named ruta-clinical-trials is included in the workspace and may serve as a starting point. The project contains the type system, an empty script and a set of 100 annotated abstracts. A simple class named Evaluate provides precision, recall and f1 score of the script applied on the given abstracts. No test data is given. The goal is to create an application that extracts the given annotations as good as possible. Furthermore, the comprehensibility, modular expandability (e.g., for other endpoints), runtime performance and maintainability of the defined rules are of importance.
